## Подтверждённая конфигурация FileZilla

- сервер работает только в активном FTP-режиме;
- число одновременных соединений FileZilla должно быть ограничено значением `1`;
- клиентский тайм-аут FileZilla нужно установить в `0` — «без тайм-аута»;
- после изменения тайм-аута FileZilla необходимо полностью закрыть и запустить снова;
- на реальном устройстве текущая сессия после изменения настройки не передавала данные, а после полного перезапуска FileZilla та же передача завершилась без ошибок;
- перезапуск FileZilla нужен для создания новой FTP-сессии; это не требует перезагрузки ZiFi или компьютера;
- клиентский тайм-аут FileZilla не связан с параметром `ftp_timeout` в `/zifi/zifi.ini`.

Это подтверждённый способ завершать текущие передачи, но не исправление
ошибок определения конца потока внутри сервера.

## Текущая архитектура загрузки на SD

- загружаемый файл сначала принимается целиком в 16 страниц по 16 КиБ;
- максимальный размер одного входящего файла — 256 КиБ;
- последовательность `STOR`: ответ `150`, открытие data-соединения, приём в RAM, закрытие data-соединения, запись на SD, ответ `226`;
- запись на SD выполняется после окончания сетевого приёма, поэтому медленная SD-карта не должна тормозить сам входящий поток;
- API 76 `APPEND` для одновременного приёма и записи не подходит: SD-карта не успевает за входящими данными;
- принято архитектурное решение: дальнейший транспорт должен использовать
  пассивный приём ESP через `AT+CIPRECVMODE=1` и явную выборку
  `AT+CIPRECVDATA`; это единственный выбранный способ создать обратное
  давление и не терять данные во время пауз SD;
- это не пассивный FTP: сервер по-прежнему работает только через `PORT/EPRT`,
  а команды FTP `PASV/EPSV` не поддерживает;
- текущая версия сервера не содержит `CIPRECVMODE/CIPRECVDATA`. На прежней
  ZiFi 1.x `CIPRECVDATA` возвращала `ERROR`, поэтому поддержку обеих команд
  нужно подтвердить на актуальном реальном модуле до переписывания транспорта.

Буфер 256 КиБ — только склад данных, а не признак конца файла. Размер `STOR`
заранее неизвестен; приём заканчивается только после строки ESP
`<DataId>,CLOSED`, которая выставляет `DataClosed`. Если `CLOSED` потерян,
сервер не отправляет `226`, сколько бы данных уже ни лежало в RAM. При файле
больше 256 КиБ сервер продолжает вычитывать поток до `CLOSED`, а затем отвечает
`552`.

## Установленные риски реализации `STOR`

1. Доказанная гонка `CLOSED`: `Cmd_STOR` вызывает `Data_Open`, а затем
   `Store_Begin`. Во время ожидания `CIPSTART` обработчик уже может принять
   `<DataId>,CLOSED` и выставить `DataClosed=1`, после чего `Store_Begin`
   безусловно стирает этот флаг нулём. `Store_Receive` остаётся ждать до
   тайм-аута, а FileZilla не получает `226`.
2. Ранний `+IPD` во время ожидания `CIPSTART` читается строковым обработчиком
   AT-ответа и не возвращается в общий двоичный парсер. Поэтому первые байты
   файла могут не попасть в RAM. Возможность потери доказана по коду;
   конкретное срабатывание на устройстве требует трассы UART.
3. Очистка очередной 16-КиБ страницы во время активного приёма создаёт длинную паузу и риск переполнения FIFO ZiFi.
4. После приёма текущий код вызывает `SAVE512` по одному сектору и отправляет `226` только после полной записи на SD. Для 256 КиБ это 512 вызовов.
5. Тесты проверяют отдельные функции, но пока не моделируют перемежение `150`, `+IPD`, `CLOSED` и `226` на реальном времени.

## Следующий системный этап

1. Проверить на актуальной прошивке реального ZiFi команды
   `AT+CIPRECVMODE=1` и `AT+CIPRECVDATA`.
2. Перевести data-сокет ESP в пассивный приём и вычитывать очередную порцию
   только тогда, когда RAM готова её принять. Не пытаться компенсировать
   скорость SD вызовами APPEND во время свободно идущего входящего потока.
3. Подготавливать RAM-буфер до открытия data-соединения.
4. Не стирать уже принятый `CLOSED`; направлять `+IPD` и `CLOSED` в общий
   парсер также во время ожидания AT-ответов.
5. Не очищать целую страницу во время сетевого приёма; обнулять только
   неполный последний сектор либо очищать память до открытия соединения.
6. После приёма писать отображённую страницу из `#C000` блоками до 32
   секторов за один `SAVE512`, проверяя результат каждого вызова. Тогда
   256 КиБ потребуют 16 вызовов вместо 512.
7. Добавить интеграционный тест полного жизненного цикла `STOR` с вариантами,
   где `+IPD` и `CLOSED` приходят до, во время и после ответа `CIPSTART`, затем
   повторить проверку на реальном устройстве.

## Локальный артефакт перед следующей аппаратной проверкой

Исходник `src/ftp_commands.asm` создаёт обычный файл с атрибутом `#00`, но локальный `ZIFIFTP.WMF` мог быть экспериментально изменён скриптом `script9.py` на атрибут `#FF`. Перед следующей проверкой на устройстве необходимо выполнить чистую сборку из текущих исходников и не включать `script9.py` в релиз.
